# テスト完了レポート: 既存物体検出モデルの導入可否評価

本レポートは、テスト計画書（[test-plan.md](./test-plan.md)）およびテスト設計書（[test-design.md](./test-design.md)）に基づいて実施した評価の結果をまとめ、受け入れ条件（AC-1〜AC-5）に対する合否と、本用途への導入可否を報告する。合格水準は Issue [#1](https://github.com/nani9ashi/qa-portfolio-object-detection/issues/1)・[#2](https://github.com/nani9ashi/qa-portfolio-object-detection/issues/2) で PdM 承認済みであり、本レポートは**事前に確定した水準に対して実測値を突き合わせた結果**を記す。実測値に合わせて合格水準を調整する操作は行っていない。

| 項目 | 内容 |
|---|---|
| 評価対象 (SUT) | YOLOv8n（COCO 事前学習・**推論のみ**。再学習・改良なし） |
| 評価データ | COCO val2017 全 5000 枚 |
| 対象クラス | person / bicycle / car / bus / truck |
| 運用点 | 信頼度スコア下限 conf 0.25、対応付け IoU 0.5（主） |
| 実行環境 | GPU（NVIDIA GeForce RTX 4080 SUPER）、ローカル推論 |
| **総合判定** | **導入不可（No-Go）** |

## 1. 総合判定

**本用途（歩行者・車両の安全補助）への現行構成の導入は「不可（No-Go）」と判定する。**

[テスト計画書](./test-plan.md) §5.2 の AND 判定方針に従い、合否対象の必須ケース（EVAL-001 / EVAL-004 / EVAL-005 / EVAL-006 / EVAL-008）のうち**一つでも合格水準を下回れば導入不可**とする。実測の結果、5 つの必須ケースのうち 4 つが不合格となった。

| 必須ケース | 観点 | 判定 |
|---|---|---|
| EVAL-001 | 全体の再現率・適合率・平均 IoU | **不合格** |
| EVAL-004 | 物体サイズ別の再現率・適合率 | **不合格** |
| EVAL-005 | 混雑度別の再現率・適合率 | **不合格** |
| EVAL-006 | 明度別の再現率・適合率 | **不合格** |
| EVAL-008 | 推論時間 | 合格 |

不合格の根本は、いずれも **AC-1 / AC-4a に関わる「再現率（見逃しの少なさ）」が合格水準を大きく下回る**点にある。適合率・位置精度（IoU）・推論時間はおおむね水準を満たしており、**本モデルの弱点は「誤検出」でも「速度」でもなく「検知漏れ」に集中している**。安全補助という用途において、見逃しは最も避けるべき誤りであり（[評価依頼書](./evaluation-request.md) §5）、この未達は導入可否を分ける決定的な要因となる。

なお、本評価における導入可否の判定は QA の評価結果としての結論であり、最終的な導入可否の意思決定は PdM が下す（[test-plan](./test-plan.md) §1.5）。本レポートはその判断材料を提供する。

## 3. 全体性能の分析（EVAL-001 / AC-1・AC-3・AC-2）

| 指標 | 実測値 | 合格水準 | 判定 |
|---|---|---|---|
| 再現率 | 0.629 | ≥ 0.95 | **不合格（−0.32）** |
| 適合率 | 0.752 | ≥ 0.70 | 合格 |
| 平均 IoU（真陽性） | 0.847 | ≥ 0.70 | 合格 |

- **再現率 0.629 は合格水準 0.95 を 0.32 ポイント下回る**。「人間オペレーター並み（再現率 95%）」を要件とする本用途（[test-plan](./test-plan.md) §5.3）に対し、現行モデルが大きく未達であることを意味する。
- 一方で**適合率 0.752・平均 IoU 0.847 は水準を満たす**。検出できた対象については、位置も妥当で誤検出も許容範囲に収まっている。
- IoU 0.75 に厳格化した補助計測では 再現率 0.516 / 適合率 0.617 と全体に低下するが、これは参考値であり合否判定には用いない（位置精度を厳格化したときの感度の観察）。

**解釈:** 本モデルの課題は「検出したものの質」ではなく「そもそも検出できていない量」にある。適合率が水準を満たす中で再現率だけが大きく未達という形は、見逃しが性能の支配的なボトルネックであることを示す。重大度は、安全に直結する見逃しが合格水準を大きく下回る **Critical**（[test-plan](./test-plan.md) §6）に該当する。

## 4. 条件別性能の分析（AC-4a / AC-4b）

条件別評価は再現率と適合率を独立した 2 軸で扱い、両方が水準を満たして初めて合格とする（[test-plan](./test-plan.md) §2.4）。各サブセットは最小要件 30 を満たし、いずれも成立した。

### 4.1 物体サイズ別（EVAL-004）

| サイズ | 再現率 | 適合率 | 正解数 | 判定 |
|---|---|---|---|---|
| small（面積 ≤ 1024px²） | **0.363** | **0.614** | 5,795 | **不合格（両軸）** |
| medium（≤ 9216px²） | **0.767** | 0.726 | 4,639 | **不合格（再現率）** |
| large（> 9216px²） | 0.904 | 0.841 | 3,272 | 合格 |

- **物体が小さいほど性能が急落する明確な勾配**が見られる。large は両軸合格だが、small は再現率 0.363（large の約 4 割）まで落ち込み、適合率も唯一 0.70 を割る。
- 小物体は全正解の 42%（5,795 / 13,706）を占めており、ここでの見逃しが全体再現率を強く押し下げている。
- 重大度 **Critical**（小物体条件で再現率が合格水準を大きく下回る）。

### 4.2 混雑度別（EVAL-005）

| 混雑度 | 再現率 | 適合率 | 画像数 | 正解数 | 判定 |
|---|---|---|---|---|---|
| 低（対象 0–2 個） | 0.816 | 0.764 | 3,505 | 1,910 | 合格 |
| 中（3–5 個） | **0.699** | 0.808 | 616 | 2,326 | **不合格（再現率）** |
| 高（6 個以上） | **0.574** | 0.734 | 879 | 9,470 | **不合格（再現率）** |

- **混雑するほど再現率が低下**する（低 0.816 → 高 0.574）。低混雑のみ両軸合格。
- 注目すべきは、**対象の 69%（9,470 / 13,706）が「高混雑」画像に集中**している点である。すなわち、実運用で守るべき対象の多くが、最も性能の落ちる条件下に存在する。全体再現率 0.629 が高混雑帯の 0.574 に近いのはこのためで、全体平均は混雑シーンの弱さに強く引きずられている。
- 適合率はいずれの混雑度でも 0.70 以上を維持しており、混雑下でも誤検出は許容範囲内。問題は一貫して見逃しにある。
- 重大度 **Critical**。

### 4.3 明度別（EVAL-006）

| 明度 | 再現率 | 適合率 | 画像数 | 判定 |
|---|---|---|---|---|
| 低（平均輝度 < 80） | **0.632** | 0.753 | 592 | **不合格（再現率）** |
| 中（80–159） | **0.628** | 0.752 | 4,061 | **不合格（再現率）** |
| 高（≥ 160） | **0.634** | 0.752 | 347 | **不合格（再現率）** |

- 全明度帯で再現率が約 0.63 と**ほぼ一定**であり、適合率も約 0.75 で揃う。すなわち**明度は本モデルの性能を分ける主要因ではない**。
- これは重要な所見である。「低照度で見逃しが増える」という事前の懸念（[test-plan](./test-plan.md) §4.3）は、本データ上は**支配的でない**。見逃しの主因は明度ではなく、§4.1・§4.2 で見たとおり**物体サイズと混雑度**にある。低照度を運用対象外にしても、全体の再現率不足は解消されない見込みである。
- なお全帯で再現率が 0.80 未満のため、AC-4a として不合格。

## 5. 参考値の傾向（EVAL-002 / EVAL-003 / EVAL-007）

### 5.1 mAP（EVAL-002）

mAP@0.5 = 0.585、mAP@0.5:0.95 = 0.405。クラス別 AP@0.5 は person 0.767 / bus 0.725 / car 0.572 / bicycle 0.471 / truck 0.391。mAP は複数 IoU 閾値を統合した参考指標であり、合否判定（単一運用点 IoU 0.5）とは粒度が異なるため参考値として記録する（[test-design](./test-design.md) §3.1）。値の水準は物体検出の一般的傾向と整合しており、評価系自体の妥当性を間接的に支持する。

### 5.2 クラス別の偏り（EVAL-003）

| クラス | 再現率 | 適合率 | 正解数 |
|---|---|---|---|
| person | 0.667 | 0.769 | 10,777 |
| car | 0.511 | 0.675 | 1,918 |
| bus | 0.650 | 0.783 | 283 |
| bicycle | 0.401 | 0.704 | 314 |
| truck | 0.353 | 0.570 | 414 |

- **person が全正解の 79%（10,777 / 13,706）を占める**。全体指標は実質的に person の性能に強く支配される。
- クラス間で再現率に大きな開き（person 0.667 と truck 0.353 で 0.31 ポイント差）があり、偏りを示すフラグが立った。**truck・bicycle・car といった車両系で再現率が低い**。
- 合格水準が「対象クラス全体」で立てられている現状の設計では、この偏りは全体平均に埋もれる。[test-design](./test-design.md) §3.1 で指摘したとおり、「歩行者と車両を検出する」という用途要件に照らせば、**クラス別（とりわけ車両系）の合格水準を別途立てるべきか**は次フェーズの論点として残る。

### 5.3 PR 曲線（EVAL-007）— 運用点変更で合格可能か

信頼度スコア下限を 0.05〜0.95 で動かしたときの全体 再現率・適合率（抜粋）:

| conf 下限 | 再現率 | 適合率 |
|---|---|---|
| 0.05 | **0.778** | 0.361 |
| 0.15 | 0.690 | 0.616 |
| 0.25（運用点） | 0.629 | 0.752 |
| 0.50 | 0.473 | 0.914 |
| 0.75 | 0.281 | 0.977 |

- **最も閾値を緩めた conf 0.05 でも、再現率の上限は 0.778 にとどまり、合格水準 0.95 には到達しない。** すなわち、**運用点（信頼度閾値）の調整だけでは AC-1 を満たすことは構造的に不可能**である。見逃しの多くは「閾値で拾えていない低スコア検出」ではなく、「そもそもモデルが出力していない対象」であることを示す。
- かつ、再現率を上げようと閾値を下げると適合率が急落する（conf 0.05 で適合率 0.361）。再現率と適合率のトレードオフは明確で、両水準（再現率 0.95 / 適合率 0.70）を同時に満たす運用点はこの曲線上に存在しない。
- この結果は、次節の対応選択肢の評価において決定的な意味を持つ。

## 6. 非機能の分析（EVAL-008 / AC-5）

| 指標 | 実測値 | 合格水準 | 判定 |
|---|---|---|---|
| 平均推論時間 | 8.1 ms / フレーム | ≤ 100 ms | 合格 |
| 中央値 / p95 | 7.7 ms / 10.8 ms | — | 参考 |

- GPU 環境（RTX 4080 SUPER）で平均 8.1 ms/フレーム（約 124 FPS 相当）。合格水準 100 ms に対して**約 12 倍の余裕**がある（ウォームアップ 10 フレーム除外、290 フレームの定常状態平均）。
- 速度は本用途のボトルネックではない。むしろこの大きな時間的余裕は、後述の対応選択肢において**より大きく重いモデルへ変更する余地が十分にある**ことを意味する。

## 7. 導入可否の結論と次フェーズへの提言

### 7.1 結論

現行構成（YOLOv8n + 運用点 conf 0.25）は、本用途の合格水準（とりわけ AC-1 再現率 95%・AC-4a 条件別再現率 80%）を満たさず、**導入不可（No-Go）**と判定する。失敗は見逃し（再現率不足）に集中しており、物体サイズが小さいほど・混雑するほど悪化する。なお明度は主因ではない。

### 7.2 対応選択肢の評価（Issue [#1](https://github.com/nani9ashi/qa-portfolio-object-detection/issues/1) で提示された A/B/C）

評価結果を、品質基準合意 Issue で議論された対応選択肢に照らして整理する。実際の変更・再評価は次フェーズで行う。

- **選択肢B（信頼度閾値の調整）— 単独では不十分。** EVAL-007 の PR 曲線が示すとおり、閾値をどう動かしても再現率は最大 0.778 で頭打ちとなり、95% に届かない。運用点調整だけで AC-1 を満たすことはできない。
- **選択肢A（より大きいモデルへの変更）— 最も有望。** 見逃しの底上げには、閾値調整ではなくモデルそのものの検出力向上が要る。EVAL-008 で推論時間に約 12 倍の余裕があるため、YOLOv8m / l 等のより大きいモデルへ切り替える余地は速度面で十分にある。次フェーズで、再現率の改善幅と推論時間（AC-5）への影響を実測して評価すべき。
- **選択肢C（用途・対象シーンの限定）— 部分導入の可能性。** large 物体（再現率 0.904）や低混雑シーン（再現率 0.816 / 適合率 0.764 で AC-4 合格）では水準近辺に達する。運用条件を「近距離・低混雑」等に限定すれば、限定的な導入余地はある。ただし小物体・混雑・全体再現率は依然未達のため、用途限定の線引きは慎重な検討を要する。なお、明度による限定（低照度を対象外とする）は §4.3 のとおり効果が薄い。

### 7.3 推奨

次フェーズでは、**選択肢 A（モデル変更）を第一候補として、再現率の改善幅と推論時間（AC-5）を実測**することを推奨する。あわせて、車両系クラスの再現率が低い点（§5.2）を踏まえ、**クラス別の合格水準を設けるかの検討**を提言する。本基準は暫定であり（Issue [#1](https://github.com/nani9ashi/qa-portfolio-object-detection/issues/1)）、事故統計に基づく絶対水準の本確定とともに、運用データを蓄積しながら見直す前提とする。最終的な導入可否および対応方針の決定は PdM が行う。

## 8. 評価の限界

本評価の結果解釈にあたり、以下の限界を明記する（[test-plan](./test-plan.md) §8）。

- **ドメイン差**: 評価は COCO の一般シーンによる事前評価であり、当社の現場映像（屋内外通路・構内）とはドメインが異なる。COCO 上の結果が現場でそのまま再現される保証はなく、現場条件での実地評価は別途必要である。
- **代表性**: COCO val2017 のシーン分布が当社現場条件をどこまで代表するかには限界がある。条件別サブセットはその限界を可視化する一助だが、完全ではない。
- **iscrowd の扱い**: 群衆として一括注釈された領域（iscrowd）は正解から除外して集計した。少数であり全体傾向への影響は小さいと判断するが、厳密な集計とは差がある。
- **単一モデル・単一解像度**: 評価対象は YOLOv8n・入力解像度 640 に限定した。モデルサイズ・解像度を変えた場合の挙動は本評価の範囲外（次フェーズ）。
- **決定性**: YOLO の推論は決定的であり、本評価は 1 回の実行で再現可能である（[test-design](./test-design.md) §5.2）。同一データ・同一モデル・同一運用点での再実行は同一結果となる。

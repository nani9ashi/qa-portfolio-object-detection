# テスト計画書: 既存物体検出モデルの導入可否評価

## 1. テスト計画の概要

### 1.1 目的

本テスト計画書は、評価依頼書（[evaluation-request.md](./evaluation-request.md)）で定義された受け入れ条件（AC-1〜AC-5）の充足確認を目的とする。調達候補である YOLO 系学習済み物体検出モデルが、当社の安全補助用途における品質要件を満たすか、公開データセットを用いて客観的に評価する。

本評価は「既存モデルの導入可否を、出力の品質から判断する」ものであり、モデルの改良・再学習は対象としない。

### 1.2 スコープ

**評価対象:**
- YOLO 系学習済みモデル（COCO 事前学習）の検出出力
- 対象クラス: person および車両系クラス（car, bicycle, bus, truck 等、COCO に含まれる範囲）

**評価対象外:**
- モデルの内部構造・学習データ等の設計（エンジニアチームの領域。QA は出力のみを評価する）
- モデルの再学習・ファインチューニング・改良
- 対象クラス以外の検出結果
- 実カメラ・撮影環境・デプロイ基盤（本評価は公開データセットによる事前評価であり、現場環境での実地評価は範囲外）

### 1.3 評価アーキテクチャ前提

本評価は以下の構成を前提とする。

```
公開データセットの画像（正解アノテーション付き）
↓
YOLO 系学習済みモデル（推論のみ）
↓
検出結果（バウンディングボックス・クラス・信頼度スコア）
↓
正解アノテーションと突き合わせ（IoU による対応付け）
↓
適合率・再現率・IoU・mAP 等を算出
↓
AC に照らして合否を判定（複数条件の AND 判定）
```

この構成により、モデルの内部に立ち入らず、入力（画像）と出力（検出結果）の対応のみで品質を測る。すなわち本評価はブラックボックス評価であり、内部構造を問わず外形的な振る舞いから品質を判断する点で、QA の標準的なテストアプローチと同じ立場をとる。

### 1.4 合格水準の確定状況

本評価で用いる合格水準について、現在の確定状況を以下に整理する。

| AC | 合格水準 | 確定状況 |
|---|---|---|
| AC-1 | 再現率 ≥ 95% | Issue（品質基準合意）で PdM 承認済み |
| AC-3 | 適合率 ≥ 70% | Issue（品質基準合意）で PdM 承認済み |
| AC-2 | 平均 IoU ≥ 0.7 | 本計画書 §5 で QA が起案、PdM 承認待ち |
| AC-4 | 条件別劣化の許容範囲 | 本計画書 §5 で QA が起案、PdM 承認待ち |
| AC-5 | 推論レスポンスの合格水準 | 本計画書 §5 で QA が起案、PdM 承認待ち |

AC-1 と AC-3 は事業判断としての品質基準のため Issue で先行合意した。残る AC-2・AC-4・AC-5 は技術的な評価設計の論点を含むため、本計画書で QA が起案し、PdM の承認を経て確定する。

### 1.5 想定体制と役割分担

| 役割 | 担当 | 主な責務 |
|---|---|---|
| PdM | 1名 | 評価依頼書作成、品質要件（特に見逃し/誤検出の重み付け）の提示、合格水準の承認、導入可否の最終判断 |
| 開発エンジニア | 1名以上 | モデルの調達・選定（本評価では実行を伴わず、評価結果を受けて次フェーズで関与する想定） |
| QA エンジニア | 1名 | 評価計画・設計・実装・実行、品質メトリクスの計測、評価結果の報告、合格水準の起案 |

本評価の遂行は QA が主担当。PdM は品質基準と合格水準の承認、最終的な導入可否の判断を担う。Dev は本評価フェーズでは実行を伴わず、評価結果に基づく対応選択肢（モデル変更・閾値調整・用途限定など）の検討は次フェーズで関与する。

## 2. 評価のアプローチ

本評価は、単一の SUT（YOLO 系モデル）の検出出力を、公開データセットの正解と突き合わせて評価する。評価の観点を以下の3層に分けて設計する。

### 2.1 全体性能の評価

評価データ全体に対する検出結果を正解と突き合わせ、適合率・再現率・mAP を算出する。これが品質判定の中心となる外形的指標である。途中の処理を問わず最終的な検出結果のみを評価する点で、システム全体をブラックボックスとして総括する。

### 2.2 条件別の性能評価

評価データを条件別サブセット（物体サイズ・混雑度・明度など）に分割し、条件ごとに性能を算出する。全体の平均値では見えない、特定条件下での性能劣化（特に見逃しの増加）を捕捉する。AC-4 に対応する。

### 2.3 誤りの非対称性の評価

見逃し（偽陰性）と誤検出（偽陽性）を分けて集計し、両者を独立に監視する。評価依頼書 §5 および品質基準合意 Issue の方針に基づき、再現率と適合率を独立した合格条件として扱い、両方を満たしてはじめて合格とする。

信頼度スコア閾値を動かしたときの再現率・適合率のトレードオフを曲線（PR 曲線）として可視化し、「片方の指標だけを最適化目標にしない」という方針を計測の形で担保する。ただし PR 曲線は判断材料として用い、合否判定は事前に合意した運用点（後述）での値で行う。

### 2.4 統合スコア（F1 等）を採用しない理由

検出性能を単一スコアに統合する指標（F1 等）は、本評価では採用しない。理由は、F1 のような統合スコアは複数モデル候補を順序付ける最適化の場では有用だが、複数の独立した合格条件を満たすかをリリース判断する本評価の目的には適さないためである。統合は片方の劣化をもう片方の余裕で隠す危険があり、安全用途では許容できない。

本評価は適合率と再現率を**独立に並べて報告し、両者がそれぞれの合格水準を満たすかを AND 判定する**設計とする。

## 3. 評価環境

| 項目 | 内容 |
|---|---|
| SUT | YOLO 系学習済みモデル（ローカル推論） |
| 実行環境 | ローカル（CPU/GPU いずれも可。本番運用は GPU 環境を想定） |
| 評価データ | COCO val2017 等の公開データセット（正解アノテーション付き） |
| 評価ライブラリ | Python（適合率・再現率・IoU・mAP の算出に標準的な評価ツールを使用） |
| API キー | 不要（外部 API 課金なし） |

ローカル推論のため、外部 API 課金が発生しない。これにより、複数の信頼度閾値・条件別サブセットでの反復算出が現実的に可能となる。ただし反復算出はあくまで PR 曲線取得や条件別評価のために行うものであり、合格水準そのものを実測値に合わせて調整するものではない。

## 4. テスト設計方針

具体的なテストケースはテスト設計書（[test-design.md](./test-design.md)）に委ね、本セクションでは設計の枠組みを示す。

### 4.1 AC と評価観点のトレーサビリティ

| AC | 内容（概要） | 対応する評価観点 |
|---|---|---|
| AC-1 | 対象物の検出（再現率） | 全体性能（再現率）/ 誤りの非対称性 |
| AC-2 | 検出位置の妥当性 | 全体性能（IoU・mAP） |
| AC-3 | 誤検出の許容範囲 | 誤りの非対称性（適合率） |
| AC-4 | 条件別の性能劣化 | 条件別性能評価 |
| AC-5 | 推論レスポンス | 非機能（推論時間計測） |

### 4.2 全体性能評価の設計方針

評価データ全体について、IoU 閾値で検出結果と正解を対応付け、真陽性・偽陽性・偽陰性を集計する。そこから適合率・再現率・mAP を算出する。IoU 閾値は主評価 0.5、補助的に 0.75 でも算出し、位置精度を厳格化したときの挙動を観察する。

### 4.3 条件別性能評価の設計方針

評価データを条件別サブセットに分割し、各サブセットで再現率を中心に算出する。全体平均では埋もれる「特定条件での見逃し増加」を明らかにする。安全用途では見逃しが最重要であるため、再現率がどの条件で落ちるかの特定を重視する。

本評価で扱う条件は、評価データの特性とサブセット作成の現実性を踏まえてテスト設計書で確定する。想定としては、COCO の標準機能で扱える**物体サイズ別**（small/medium/large）が確実に扱える範囲。**混雑度**（画像内の対象クラス数による分類）はアノテーションから自動算出できるため扱える見込み。**明度**（低照度）は画像のピクセル値の平均明度から自動分類できるが、COCO の画像分布における低照度画像の十分性については要確認。**遮蔽**は人手判定が要るため本評価では範囲外とする可能性がある。

### 4.4 誤りの非対称性評価の設計方針

信頼度スコアの下限を複数設定し、各設定での再現率・適合率を算出して、両者のトレードオフを PR 曲線として描く。これにより、「再現率を上げるために閾値を下げると、誤検出がどれだけ増えるか」を定量的に示す。本曲線は合否判定の直接的根拠ではなく、運用点選択の判断材料として用いる。

合否判定は、事前に合意した運用点（信頼度スコア下限）における再現率と適合率の値で行う。運用点の初期値はテスト設計書で起案する。

### 4.5 評価ケースの優先度定義

| 優先度 | 判定基準 | 例 |
|---|---|---|
| 高 | 安全に直結する観点。失敗が見逃しや重大な誤作動を示す。 | 対象クラスの再現率、条件別の見逃し増加 |
| 中 | 運用品質に関わる観点。 | 適合率（誤検出率）、IoU の位置精度、推論時間 |
| 低 | 補助的・参考的な観点。 | 厳格 IoU（0.75）での参考値、クラス別の細かい内訳 |

## 5. 品質メトリクスと合否基準

本評価で用いる合格水準を以下に定める。AC-1 および AC-3 は Issue（品質基準合意）で PdM 承認済み、AC-2・AC-4・AC-5 は本計画書で QA が起案する。

### 5.1 メトリクスと合格水準

| メトリクス | 計測方法 | 合格水準 | 対応 AC | 確定状況 |
|---|---|---|---|---|
| 再現率（対象クラス全体） | 運用点 conf 下限で正解と対応付け、偽陰性を集計 | **≥ 95%** | AC-1 | PdM 承認済み（Issue） |
| 適合率（対象クラス全体） | 同上、偽陽性を集計 | **≥ 70%** | AC-3 | PdM 承認済み（Issue） |
| 平均 IoU（真陽性検出） | 検出と正解の重なり | ≥ 0.7 | AC-2 | QA 起案、PdM 承認待ち |
| mAP@0.5 | クラス別 AP の平均（IoU 0.5） | 参考値として併記 | AC-1/AC-2 | 参考指標 |
| mAP@0.5:0.95 | IoU を厳格化した平均 | 参考値として併記 | AC-1/AC-2 | 参考指標 |
| 条件別の再現率 | 各サブセットで算出 | 全体再現率の **80% 以上を維持** | AC-4 | QA 起案、PdM 承認待ち |
| 推論時間 | 1 フレームあたり | **≤ 100ms（GPU 運用環境）** | AC-5 | QA 起案、PdM 承認待ち |
| PR 曲線 | 信頼度閾値を動かして算出 | 判断材料として提示（合否対象外） | AC-1/AC-3 | 参考指標 |

### 5.2 合否判定の方式

本評価の合否は、複数の独立した合格条件を **AND 判定** で扱う。すなわち、上記の合格水準のうち一つでも下回るものがあれば、本用途への導入は**不可**と判定する。F1 のような統合スコアによる丸めは行わない（§2.4 参照）。

ただし「参考指標」とした mAP および PR 曲線は、合否判定の直接的対象ではなく、評価結果を解釈するための補助情報として用いる。

### 5.3 起案数値の根拠

**平均 IoU ≥ 0.7（AC-2）**: 検出位置の重なりが 0.7 を下回ると、対象の位置を正しく伝えるという用途上の意味が損なわれる。現場担当者に「この辺りに対象あり」と注意喚起する用途として、位置精度 0.7 は最低限必要な水準と判断した。

**条件別の再現率: 全体の 80% 以上維持（AC-4）**: 特定条件下で再現率が大きく劣化することは、その条件下では本機能が安全補助として機能しないことを意味する。「全体水準の 80% 以上を維持」という形で歯止めをかけ、これを下回る条件は「用途として対応不可」と明確化する基準とした。

**推論時間 ≤ 100ms（AC-5）**: 本機能は連続フレーム処理による補助を想定する。歩行者・車両は秒単位で移動するため、検出から注意喚起までの遅延は 100ms（=10FPS）以内に収めることが、移動に追従するための実用的な水準である。本番運用は GPU 環境を想定しており、YOLOv8n 級のモデルなら GPU で 100ms 以内は十分達成可能な範囲。

これらは QA としての起案である。PdM の承認をもって確定とする。

## 6. 評価上の問題の分類

評価で検出された問題を、影響の観点から分類する。

| 重大度 | 定義 | 例 |
|---|---|---|
| Critical | 安全に直結する見逃しが、合格水準を大きく下回る | 主要条件で対象クラスの再現率が合格水準を大きく下回る |
| Major | 特定条件での顕著な性能劣化、または誤検出の過多 | 低照度で再現率が大きく低下、適合率が下限を下回る |
| Minor | 補助指標の軽微な未達 | 厳格 IoU での位置精度がやや不足 |

重大度はあくまで本用途（安全補助）を前提とした分類であり、別用途では見逃しと誤検出の重み付けが変わるため、分類も変わりうる。最終的な判断は PdM と協議する。

## 7. 評価終了条件

以下を満たした時点で評価完了とみなす。

- 全評価観点について算出が完了し、AC との対応が取れている
- §5 で定めた合格水準すべてに対して、合否判定が下されている
- 「不可」と判定された AC がある場合、その原因が分析・記録されている
- 評価結果に基づく導入可否の判断が PdM によって下されている

## 8. リスクと対応方針

| リスク | 影響度 | 対応方針 |
|---|---|---|
| COCO 学習済みモデルと当社現場映像のドメイン差により、公開データでの評価結果が現場で再現されない | 高 | 本評価が公開データによる事前評価である旨を明示し、現場での実地評価を別途提言する |
| 評価用公開データのシーンが、当社現場条件を十分代表していない | 中 | 条件別サブセットで代表性の限界を可視化し、結果解釈に明記する |
| 合格水準を QA 単独で設定し、用途のリスク許容度とずれる | 中 | 主要な合格水準（AC-1・AC-3）は Issue で PdM 承認を経て確定。その他（AC-2・AC-4・AC-5）も本計画書で起案し PdM 承認を経る |
| 統合スコア（F1 等）を求められたときに、本評価の AND 判定方針との整合をどう保つか | 低 | テスト完了レポートで F1 不採用の理由を明示。最適化の場と意思決定の場の指標の使い分けとして整理する |
